# 第 1 章：现有主链与插入点

这一章只做一件事：让你知道新功能应该插在哪，不要把 LightRAG 和 MCP 硬塞进错误的层。

## 1. 现在的主链长什么样

你当前仓库已经有这条主链：

```text
runtime.py -> graph_runtime.py -> ToolNode -> loop.py
```

它的职责应该保持清晰：

- `ipilot/runtime.py` 负责总装配
- `ipilot/agent/graph_runtime.py` 负责图和工具调度
- `ipilot/agent/langchain_tools.py` 负责本地工具
- `ipilot/agent/loop.py` 负责对外 facade

## 2. 为什么不在 `loop.py` 里加 LightRAG

`loop.py` 的任务只是：

- 接收用户消息
- 调用 graph
- 整理返回值

它不应该知道：

- 哪个工具来自 MCP
- 哪个工具来自本地文件系统
- LightRAG 怎么建库
- MCP server 怎么连

如果你把这些都塞进去，后面一定位不好问题。

## 3. 为什么不直接改 `graph_runtime.py`

`graph_runtime.py` 已经做对了一件事：它只负责把模型和工具串起来。

第一版接入 MCP 时，最稳的做法是：

1. 先把远端 MCP tools 变成本地可调用的 LangChain tools
2. 再把这些 tools 一起喂给 `build_agent_graph(...)`

这样 `graph_runtime.py` 基本不用动。

## 4. 这次真正要新增的层

你会新增三层东西：

- `services/lightrag_mcp/`：独立 LightRAG MCP server
- `ipilot/mcp/`：通用 MCP client 和工具桥接
- `ipilot/agent/toolset.py`：本地工具 + MCP 工具的聚合层

## 5. 推荐的落点顺序

照着这个顺序做最不容易乱：

1. 先写 `services/lightrag_mcp/`
2. 再写 `ipilot/mcp/`
3. 再写 `ipilot/agent/toolset.py`
4. 最后改 `runtime.py`
